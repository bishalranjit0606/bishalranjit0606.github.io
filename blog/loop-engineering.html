Loop Engineering: Automated Prompt Engineering

Prompt engineering is one message, one answer. Loop engineering is when the system keeps going on its own: trigger, run, check, save state, repeat or stop. Think of it like cruise control for your AI agent.

1. What is loop engineering?

Loop engineering is just automated prompt engineering. Instead of you typing the next prompt every time, you design a system that does it for you.

A normal loop is simple:

Trigger → Action → Stop

A real AI loop has more steps:

Trigger → Execute → Verification → State → Trigger (again)

You stop being the person who keeps prompting. You become the person who designs the loop. For a deeper visual guide, see Loop Engineering with Mayank.

2. Prompt engineering vs loop engineering

  • Prompt engineering: you write one instruction, AI replies, you check, you write again.
  • Loop engineering: you set the goal, steps, checks, and stop rules once. The system runs the loop.

Prompt engineering still matters. Every loop still uses prompts inside it. Loop engineering is about the system around those prompts.

3. JSON prompt vs loop JSON

They look almost the same. The big difference is the goal section.

Prompt engineering JSON (one shot):

{
  "task": "Write a LinkedIn post",
  "topic": "Saving AI tokens",
  "tone": "simple and practical",
  "length": "short",
  "avoid": "emoji and complex words"
}

Loop engineering JSON (same idea, but goal has clear pass/fail rules):

{
  "goal": {
    "task": "Write a LinkedIn post",
    "topic": "Trending AI news today",
    "done_when": {
      "word_count": "120 to 180",
      "readability": "grade 8 or lower",
      "must_include": "one real tip",
      "must_avoid": "emoji, buzzwords"
    }
  },
  "trigger": "every weekday at 7am",
  "execution": "research trends, draft post",
  "verification": {
    "type": "judgment",
    "pass_score": "7 out of 10 or higher",
    "max_attempts": 3
  },
  "state": "save each draft, score, and what worked before",
  "stop": "stop after 3 tries or when score passes"
}

Same JSON style. But loop JSON adds done_when, verification, state, and stop. That is the whole point.

4. The 4 pieces of every loop

  1. Trigger - what starts it? A schedule, a new email, a GitHub event, or you typing a command.
  2. Execution - the agent does the work (research, write code, draft a post).
  3. Verification - did it actually finish well? This is the most important part.
  4. State - what happened in past loops? Save it so the next run is smarter.

There is also a 5th piece: stop condition. Without it, the loop keeps running and burns tokens.

4 types of verification

  • Functional - machine can check. Example: tests pass, build succeeds.
  • Visual - someone or AI must look. Example: does the UI match the design?
  • Judgment - needs a rubric and score. Example: rate this post 1 to 10.
  • You decide - human must approve. Example: should we send this email?

5. Verification and state are the main things

Humans are good at saying what they want. The loop needs that same clarity.

Do not say:

Too vague

Make the article better.

Say what "better" means:

Clear

Rewrite until readability is grade 8 or lower, word count is 120 to 180, and it includes at least one real tip.

State means the loop remembers past runs. Example: a LinkedIn loop saves old posts and which ones got good engagement. Next time it writes based on that data, not from zero.

6. Use numbers, not feelings

Good loops use metrics. Bad loops use vague words.

  • Word count: 120 to 180 words
  • Test pass rate: 100% of tests green
  • Visual match score: 8 out of 10 or higher
  • Runtime improvement: at least 15% faster than last version
  • Max attempts: 3 tries, then stop

Numbers make verification possible. "Better" and "faster" alone do not.

7. How to build a loop (5 steps)

Nobody starts with a full loop. This is the real path:

Step 1: Do it manually
   ↓
Step 2: Turn it into a skill
   ↓
Step 3: Automate the trigger
   ↓
Step 4: Add verification + state
   ↓
Step 5: You have a loop
  • Step 1: Do the task by hand. Confirm AI can actually do it.
  • Step 2: Save the steps as a skill or template so you do not rewrite prompts.
  • Step 3: Add a trigger (cron job, schedule, webhook).
  • Step 4: Add a real check and save results from each run.
  • Step 5: Connect it all. Now it loops on its own.

8. Should you even build one?

Loops cost money (tokens, time, setup). Not every task needs one.

Build a loop when:

  • The task repeats (daily posts, inbox triage, CI fixes).
  • It has many steps you cannot predict in one prompt.
  • You can define a real "done" check (tests, score, word count).

Just use a normal prompt when:

  • It is a one-time task.
  • You can finish it in one or two messages.
  • There is no clear way to check if it is done.

Always set a max number of tries. Example: stop after 3 attempts or 10 minutes. This saves tokens.

9. What to skip for now

  • 24/7 swarms of agents prompting other agents
  • Loops with no stop limit
  • Trusting "looks done" with no real check
  • Looping a simple one-shot task you could just prompt once

Start with one small repeatable task. Pick the right check type. Set a max try count. That is enough.

Final summary

Loop engineering is not magic. It is automated prompt engineering with structure.

  • Prompt = one shot. Loop = trigger, execute, verify, state, repeat or stop.
  • JSON for loops looks like prompt JSON, but the goal must say exactly when it is done.
  • Verification and state are the heart of a good loop.
  • Use numbers (word count, score, pass rate), not vague words like "better."
  • Build step by step: manual → skill → trigger → verification + state → loop.
  • Always set max tries and a stop condition to save tokens.
