# Simple English

Use this style in every reply and in all text you write for this project.

## Core rules

1. Use short sentences.
2. Use common words that most people know.
3. Do not use fancy or formal words when a simple word works.
4. Do not use dashes in prose. This includes em dashes, en dashes, and dash pairs used to add extra detail.

## Words to avoid

Replace hard words with simple ones:

| Avoid | Use instead |
|-------|-------------|
| utilize | use |
| facilitate | help |
| implement | add, build, set up |
| leverage | use |
| optimize | improve, fix |
| subsequently | then, after that |
| furthermore | also |
| approximately | about |
| demonstrate | show |
| functionality | feature, what it does |

## Dash examples

Bad: "We improved efficiency — reducing API calls by half."

Good: "We improved speed. We cut API calls in half."

Bad: "The update — which ships today — fixes the bug."

Good: "The update ships today. It fixes the bug."

Bad: "Fast, reliable, secure — all in one tool."

Good: "The tool is fast. It is reliable. It is secure."

## How to rewrite dash sentences

Split one long sentence into two or three short ones.

Use a period instead of a dash.

Use "and", "but", "so", or "because" when you need to connect ideas.

## UI and docs copy

Write like you talk to a friend who is learning English.

Say what the user should do in plain words.

Keep buttons and labels short: "Save changes", not "Persist modifications".

Errors should say what went wrong and what to do next.

## When dashes are OK

Hyphens inside code, file names, URLs, and technical terms are fine. Examples: `user-id`, `README.md`, `npm install`.

Minus signs in numbers and math are fine. Example: -5, 3-2=1.

Bullet lists with `-` are fine in markdown when the project needs them.

## Before you send

Read your text once.

Remove any dash used to join two ideas in prose.

Replace any word that feels hard or formal.

Shorten long sentences.
